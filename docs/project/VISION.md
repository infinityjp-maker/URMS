## URMS が存在する理由

資産（物理資産・デジタル資産・人的資産・知識資産）を **統合的に管理・可視化・最適化** する必要がある。  
散在した台帳、属人化した運用、短期導線のシステムでは、10年スパンで価値を失う。

URMS は **単一の信頼できる資産の正本（SSOT）** となり、組織の意思決定と運用を支える。

---

## 最終目標

**10年以上保守できる、世界最高レベルの資産統合管理システム** を実現する。

- 利用者にとって直感的で誤りにくい
- 開発者にとって変更しやすく、恐れない
- 運用者にとって監査可能で安全
- AI と人間が協調して進化できる

---

## 設計思想

1. **SSOT（Single Source of Truth）** — 同じ事実を複数箇所に書かない
2. **境界の明確化** — Prisma（ORM）と PostgreSQL（DB 本体）の責務分離
3. **承認ゲート** — 設計・実装・完成・知識更新に明示的な承認
4. **ドキュメント駆動** — コードより先に判断と記録
5. **最小複雑性** — 必要になるまで抽象化・最適化しない
6. **拡張可能な構造** — 将来 URMS 自身が AI チームを Resource として管理

---

## 判断基準

迷ったときは以下の順で判断する。

1. **10年後も正しいか** — 短期効率より長期保守性
2. **SSOT を壊していないか** — 複製・矛盾の増殖を避ける
3. **承認を経ているか** — PM / Architect / Reviewer / Tester / KM
4. **AI と人間の両方が理解できるか** — 可読性・一貫性・日本語品質
5. **セキュリティと監査可能性** — 後回しにしない

---

## AI チームが共有すべき理念

- **PM 単一窓口** — User との直接会話は PM のみ
- **ロール尊重** — 自分の領域外を侵さない
- **引き継ぎの明示** — Context と docs で「今どこにいるか」を共有
- **Knowledge は KM** — 長期記憶の正本は `docs/project/`
- **勝手に実装しない** — PM + Architect 承認なしの実装禁止
- **日本語で明確に** — 報告・ドキュメント・ADR

---

## 10年以上保守するための哲学

| 原則 | 意味 |
|------|------|
| **変更容易性** | 小さな変更が大きな破壊を招かない構造 |
| **追跡可能性** | ADR と architecture-history で「なぜ」を残す |
| **テスト可能性** | 重要ロジックは Tester ゲートを通す |
| **人の入れ替え** | ドキュメントと用語集で属人化を防ぐ |
| **AI の入れ替え** | Rules / Skills / Commands で行動を固定 |
| **技術的負債の可視化** | backlog と ADR で負債を隠さない |
