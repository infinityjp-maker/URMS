# CI 修復の背景と設計判断

目的: URMS の「構造美」「合理性」「長期運用性」に沿って、今回の CI 修復で取ったアプローチとその理由を文書化します。

1) 最小差分主義

 - 理由: 既存のテストロジックやアプリケーションコードへの影響を最小化し、デバッグコストを抑えるため。CI 固有の補助（example: `CI_BROWSER_ARGS`）を用いることで、本体の挙動を壊さずに環境差を吸収します。

2) 可逆性と明示的フォールバック

 - 理由: CI 環境は多様（hosted runner、セルフホスト、コンテナ内など）なため、各環境で試行→失敗→別ルートの実行ができるようフォールバックを明示しました（/etc/hosts の失敗耐性、Docker の host-net フォールバック、Docker 無しの直接実行）。

3) 診断を最重要視

 - 理由: 失敗時の迅速なトリアージのために `builds/diagnostics/` に一貫した出力を残すことを重視しました。故障の根本原因（CORS / 名前解決 / ネットワーク名前空間）を素早く切り分けられるようにしています。

4) 安全性と最小権限

 - 理由: `/etc/hosts` の書き換えや sudo 権限に依存する実装は最小化し、失敗しても継続する設計にしました。将来はブラウザ側の解決ルール (`host-resolver-rules`) を優先する方向を推奨します。

5) ドキュメント化による継続的改善

 - 理由: 今回の変更は「一時的な環境差への対応」以上に、将来的な CI 安定化の設計指針となるべきです。今回のドキュメントは次の担当者が容易に運用改善できるように作成しました。
