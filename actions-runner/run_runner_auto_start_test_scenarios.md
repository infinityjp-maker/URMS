run_runner_auto_start.cmd — 起動テストシナリオ

1) 二重起動時の期待挙動
- 前提: 既に `runner_auto_start.pid` が存在し、かつその PID のプロセスが稼働中。
- 期待: `run_runner_auto_start.cmd` を再実行すると即座に終了し、既存デーモンは継続して稼働する。ログには「already running」などの情報が記載される。

2) PID ファイル破損時の挙動
- 前提: `runner_auto_start.pid` が存在するが内容が破損（非数値）または古い PID を指す。
- 期待: ラッパーは PID を読み取り失敗した場合は古い PID ファイルを削除して新プロセスを起動する。

3) 実行ポリシー例外時の挙動
- 前提: システムで `AllSigned` 等によりスクリプト実行が制約されている。
- 期待: ラッパーは `-ExecutionPolicy Bypass` を指定して `pwsh` を起動するため、通常は起動成功する。
- 例外: 環境ポリシーで全ての外部起動が禁止されている場合はログにエラーを記録し動作終了する。

4) run.cmd が見つからない場合
- 期待: スクリプトは 10 秒待機してリトライし、`run.cmd` が所定パスに無ければ継続的にリトライする（明記されたログ出力あり）。

5) Runner.Listener の検出/終了
- 期待: run.cmd 起動後、`Runner.Listener` プロセスの出現を検知し、`Wait-Process` で終了を待ち、終了次第ループに戻る。

各シナリオは実稼働前にログと PID の変化を人が確認して承認すること。
